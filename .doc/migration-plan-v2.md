### 1.4 유스케이스 (Use Cases)

#### UC-001: 키오스크 주문

| 항목              | 내용                                 |
| ----------------- | ------------------------------------ |
| **Actor**         | 고객                                 |
| **Precondition**  | 키오스크 메인 화면(웰컴 스크린) 표시 |
| **Postcondition** | 주문 완료 및 영수증 출력             |

**Main Flow:**

1. 고객이 언어를 선택한다 (한/영/일/중)
2. 고객이 카테고리를 선택한다
3. 고객이 상품을 선택한다
4. (선택 사항) 고객이 옵션을 선택한다
5. 고객이 수량을 지정한다
6. 고객이 장바구니에 담기를 선택한다
7. (반복) 2-5단계를 추가 상품에 대해 반복
8. 고객이 결제하기를 선택한다
9. 고객이 결제 수단을 선택한다 (카드/현금)
10. 시스템이 결제를 처리한다
11. 시스템이 영수증을 출력한다
12. 주문 완료 화면이 표시된다

**Alternative Flow:**

| 단계 | 조건                      | 대체 흐름                                   |
| ---- | ------------------------- | ------------------------------------------- |
| 3a   | 상품이 품절인 경우        | "품절" 표시, 선택 불가                      |
| 4a   | 옵션 상품인 경우          | 옵션 선택 모달 표시                         |
| 9a   | 카드 잔액 부족            | "잔액 부족" 메시지, 8단계로 이동            |
| 9b   | VAN 타임아웃              | 자동 재시도 (최대 3회), 실패 시 에러 메시지 |
| 9c   | 네트워크 장애 (현금 결제) | 오프라인 모드로 처리, 동기화 대기열에 추가  |
| 9d   | 네트워크 장애 (카드 결제) | "카드 결제 불가" 메시지, 현금 결제 안내     |

#### UC-002: 결제 취소

| 항목              | 내용                       |
| ----------------- | -------------------------- |
| **Actor**         | 관리자                     |
| **Precondition**  | 당일 결제 완료된 거래 존재 |
| **Postcondition** | 결제 취소 및 환불 처리     |

**Main Flow:**

1. 관리자가 관리자 모드로 진입한다
2. 관리자가 거래 내역을 조회한다
3. 관리자가 취소할 거래를 선택한다
4. 관리자가 취소 사유를 입력한다
5. 시스템이 VAN에 취소 요청을 전송한다
6. 시스템이 취소 영수증을 출력한다

**Alternative Flow:**

| 단계 | 조건                  | 대체 흐름                    |
| ---- | --------------------- | ---------------------------- |
| 3a   | 당일 거래가 아닌 경우 | "당일 취소만 가능" 메시지    |
| 5a   | 취소 실패             | 에러 메시지, VAN사 문의 안내 |

#### UC-003: 회원 포인트 적립/사용

| 항목              | 내용                                 |
| ----------------- | ------------------------------------ |
| **Actor**         | 고객                                 |
| **Precondition**  | 장바구니에 상품 존재, 결제 단계 진입 |
| **Postcondition** | 포인트 적립 또는 사용 완료           |

**Main Flow (적립):**

1. 고객이 "회원 적립" 버튼을 선택한다
2. 고객이 전화번호를 입력한다
3. 시스템이 회원 정보를 조회한다
4. 시스템이 회원 이름을 표시한다
5. 고객이 확인을 선택한다
6. 결제 완료 시 포인트가 적립된다

**Main Flow (사용):**

1. 고객이 "포인트 사용" 버튼을 선택한다
2. 고객이 전화번호를 입력한다
3. 시스템이 회원 정보와 포인트 잔액을 조회한다
4. 고객이 사용할 포인트 금액을 입력한다
5. 시스템이 결제 금액에서 차감한다

**Alternative Flow:**

| 단계 | 조건             | 대체 흐름                          |
| ---- | ---------------- | ---------------------------------- |
| 3a   | 회원 미등록      | "등록되지 않은 회원" 메시지        |
| 3b   | 네트워크 장애    | "회원 조회 불가" 메시지, 적립 불가 |
| 4a   | 포인트 잔액 부족 | 사용 가능 금액만 표시              |

#### UC-004: 오프라인 모드 운영

| 항목              | 내용                                      |
| ----------------- | ----------------------------------------- |
| **Actor**         | 시스템, 고객                              |
| **Precondition**  | 네트워크 연결 끊김 감지                   |
| **Postcondition** | 오프라인 주문 처리, 온라인 복구 시 동기화 |

**Main Flow:**

1. 시스템이 네트워크 연결 끊김을 감지한다
2. 시스템이 오프라인 모드로 전환한다
3. 화면에 "오프라인 모드" 표시
4. 고객이 상품을 선택하고 주문한다
5. 고객이 현금 결제를 선택한다
6. 시스템이 로컬 DB에 주문을 저장한다
7. 영수증이 출력된다 (오프라인 표시 포함)

**온라인 복구 시:**

1. 시스템이 네트워크 연결 복구를 감지한다
2. 시스템이 동기화 대기열의 주문을 서버로 전송한다
3. 서버가 주문을 처리하고 응답한다
4. 충돌 발생 시 Conflict Resolution 적용

**Alternative Flow:**

| 단계 | 조건                 | 대체 흐름                                 |
| ---- | -------------------- | ----------------------------------------- |
| 5a   | 카드 결제 시도       | "오프라인 모드에서 카드 결제 불가" 메시지 |
| 6a   | 로컬 큐 초과 (100건) | "더 이상 주문 불가" 메시지, 관리자 호출   |

---

#### UC-005: 영수증 재출력

| 항목              | 내용               |
| ----------------- | ------------------ |
| **Actor**         | 관리자             |
| **Precondition**  | 이전 거래 존재     |
| **Postcondition** | 영수증 재출력 완료 |

**Main Flow:**

1. 관리자가 관리자 모드로 진입한다
2. 관리자가 "영수증 재출력"을 선택한다
3. 관리자가 거래를 검색한다 (날짜, 주문번호)
4. 관리자가 재출력할 거래를 선택한다
5. 시스템이 영수증을 출력한다 (재출력 표시)

```typescript
import Dexie, { Table } from "dexie";

interface OfflineOrder {
  id?: number;
  localId: string; // 로컬 생성 UUID
  serverId?: string; // 동기화 후 서버 ID
  items: OrderItem[];
  totalAmount: number;
  paymentType: "CASH" | "PENDING_CARD";
  status: "PENDING_SYNC" | "SYNCED" | "CONFLICT";
  createdAt: Date;
  syncedAt?: Date;
}

interface OfflineProduct {
  id: number;
  barcode: string;
  name: string;
  sellPrice: number;
  categoryId: number;
  imageUrl?: string;
  lastSyncedAt: Date;
}

class OfflineDatabase extends Dexie {
  orders!: Table<OfflineOrder>;
  products!: Table<OfflineProduct>;
  syncQueue!: Table<SyncQueueItem>;

  constructor() {
    super("poson-offline");
    this.version(1).stores({
      orders: "++id, localId, serverId, status, createdAt",
      products: "id, barcode, categoryId, lastSyncedAt",
      syncQueue: "++id, type, entityId, action, createdAt",
    });
  }
}

export const offlineDb = new OfflineDatabase();
```

#### 4.2.1 부분 환불 비즈니스 규칙

```typescript
// src/services/payment/partial-refund.service.ts

interface PartialRefundRequest {
  originalTransactionId: string;
  refundAmount: number;
  reason: RefundReason;
  items?: RefundItem[]; // 품목별 환불 시
}

interface RefundValidationResult {
  valid: boolean;
  errorCode?: string;
  errorMessage?: string;
}

/**
 * 부분 환불 비즈니스 규칙 (P2)
 */
class PartialRefundService {
  /**
   * 부분 환불 요청 검증
   */
  validateRefund(request: PartialRefundRequest, original: Transaction): RefundValidationResult {
    // 1. 원거래 상태 확인
    if (original.status !== "CAPTURED") {
      return {
        valid: false,
        errorCode: "INVALID_STATUS",
        errorMessage: "결제 완료(CAPTURED) 상태에서만 환불 가능합니다",
      };
    }

    // 2. 환불 기간 확인 (7일 이내)
    const daysSinceCapture = this.getDaysSince(original.capturedAt);
    if (daysSinceCapture > 7) {
      return {
        valid: false,
        errorCode: "REFUND_PERIOD_EXCEEDED",
        errorMessage: "환불 가능 기간(7일)이 초과되었습니다",
      };
    }

    // 3. 누적 환불 금액 확인
    const totalRefunded = this.getTotalRefundedAmount(original.id);
    const remainingRefundable = original.amount - totalRefunded;

    if (request.refundAmount > remainingRefundable) {
      return {
        valid: false,
        errorCode: "EXCEEDS_REFUNDABLE",
        errorMessage: `환불 가능 금액(${remainingRefundable}원)을 초과합니다`,
      };
    }

    // 4. 최소 환불 금액 확인
    if (request.refundAmount < 100) {
      return {
        valid: false,
        errorCode: "BELOW_MINIMUM",
        errorMessage: "최소 환불 금액은 100원입니다",
      };
    }

    // 5. 환불 횟수 제한 (동일 거래 최대 5회)
    const refundCount = this.getRefundCount(original.id);
    if (refundCount >= 5) {
      return {
        valid: false,
        errorCode: "MAX_REFUND_COUNT",
        errorMessage: "부분 환불 횟수 제한(5회)을 초과했습니다",
      };
    }

    return { valid: true };
  }

  /**
   * 부분 환불 처리 후 상태 결정
   */
  determineStatusAfterRefund(original: Transaction, refundAmount: number): TransactionStatus {
    const totalRefunded = this.getTotalRefundedAmount(original.id) + refundAmount;

    if (totalRefunded >= original.amount) {
      return "REFUNDED"; // 전액 환불 완료
    } else {
      return "PARTIAL_REF"; // 부분 환불 상태 유지
    }
  }
}
```

| 규칙        | 조건                  | 에러 코드              |
| ----------- | --------------------- | ---------------------- |
| 원거래 상태 | CAPTURED만 허용       | INVALID_STATUS         |
| 환불 기간   | 결제 완료 후 7일 이내 | REFUND_PERIOD_EXCEEDED |
| 누적 금액   | 원거래 금액 초과 불가 | EXCEEDS_REFUNDABLE     |
| 최소 금액   | 100원 이상            | BELOW_MINIMUM          |
| 횟수 제한   | 동일 거래 최대 5회    | MAX_REFUND_COUNT       |

```typescript
  ): Promise<PaymentResult> {
    // 1. 기존 처리 결과 확인
    const existing = await this.redis.get(`payment:idem:${idempotencyKey}`);
    if (existing) {
      const record: IdempotencyRecord = JSON.parse(existing);
      if (record.status === "COMPLETED" && record.result) {
        return record.result; // 기존 결과 반환
      }
      if (record.status === "PROCESSING") {
        throw new PaymentInProgressError("결제가 처리 중입니다");
      }
    }

    // 2. 처리 시작 표시
    await this.redis.setex(
      `payment:idem:${idempotencyKey}`,
      this.TTL,
      JSON.stringify({ status: "PROCESSING", createdAt: new Date() }),
    );

    try {
      // 3. 결제 처리
      const result = await paymentFn();

      // 4. 결과 저장
      await this.redis.setex(
        `payment:idem:${idempotencyKey}`,
        this.TTL,
        JSON.stringify({
          status: "COMPLETED",
          result,
          createdAt: new Date(),
        }),
      );

      return result;
    } catch (error) {
      // 5. 실패 기록
      await this.redis.setex(
        `payment:idem:${idempotencyKey}`,
        this.TTL,
        JSON.stringify({
          status: "FAILED",
          error: error.message,
          createdAt: new Date(),
        }),
      );
      throw error;
    }
  }
}
```

#### 4.5.2 암호화 구현

```typescript
// src/services/payment/encryption.service.ts
import { KMSClient, DecryptCommand, EncryptCommand } from "@aws-sdk/client-kms";
import * as crypto from "crypto";

interface EncryptedData {
  ciphertext: string; // Base64 인코딩된 암호문
  iv: string; // 초기화 벡터
  authTag: string; // GCM 인증 태그
  keyVersion: string; // 키 버전 (키 로테이션 추적)
}

class PaymentEncryptionService {
  private kms: KMSClient;
  private readonly KEY_ID = process.env.KMS_PAYMENT_KEY_ID;
  private readonly ALGORITHM = "aes-256-gcm";

  constructor() {
    this.kms = new KMSClient({ region: process.env.AWS_REGION });
  }

  /**
   * 민감 결제 데이터 암호화
   * - AES-256-GCM 알고리즘 사용 (인증된 암호화)
   * - AWS KMS로 데이터 키 보호
   */
  async encrypt(plaintext: string): Promise<EncryptedData> {
    // 1. KMS에서 데이터 키 생성
    const dataKey = await this.generateDataKey();

    // 2. 초기화 벡터 생성 (96비트 권장)
    const iv = crypto.randomBytes(12);

    // 3. AES-256-GCM 암호화
    const cipher = crypto.createCipheriv(this.ALGORITHM, dataKey.plaintext, iv);

    let ciphertext = cipher.update(plaintext, "utf8", "base64");
    ciphertext += cipher.final("base64");

    const authTag = cipher.getAuthTag();

    // 4. 평문 데이터 키는 메모리에서 즉시 삭제
    dataKey.plaintext.fill(0);

    return {
      ciphertext,
      iv: iv.toString("base64"),
      authTag: authTag.toString("base64"),
      keyVersion: dataKey.version,
    };
  }

  /**
   * 민감 결제 데이터 복호화
   */
  async decrypt(encrypted: EncryptedData): Promise<string> {
    // 1. KMS에서 데이터 키 복호화
    const dataKey = await this.decryptDataKey(encrypted.keyVersion);

    // 2. AES-256-GCM 복호화
    const decipher = crypto.createDecipheriv(
      this.ALGORITHM,
      dataKey,
      Buffer.from(encrypted.iv, "base64"),
    );

    decipher.setAuthTag(Buffer.from(encrypted.authTag, "base64"));

    let plaintext = decipher.update(encrypted.ciphertext, "base64", "utf8");
    plaintext += decipher.final("utf8");

    // 3. 데이터 키 메모리에서 삭제
    dataKey.fill(0);

    return plaintext;
  }

  /**
   * 카드번호 마스킹 (로그/표시용)
   * 예: 1234-5678-9012-3456 → 1234-****-****-3456
   */
  maskCardNumber(cardNumber: string): string {
    const cleaned = cardNumber.replace(/[-\s]/g, "");
    if (cleaned.length < 13) return "****";
    return `${cleaned.slice(0, 4)}-****-****-${cleaned.slice(-4)}`;
  }
}
```

```typescript
   * 변경된 파일 기반 영향도 분석
```

#### 6.4.1 재해 유형별 분류

| 재해 유형              | 심각도   | RTO 목표 | 복구 절차                  |

| **VAN 연결 장애**      | Medium   | 5분      | Failover VAN 전환          |

#### 6.4.2 DR 시나리오별 복구 절차

```yaml
# 시나리오 1: 데이터베이스 장애
database_failure:
  detection:
    - CloudWatch RDS 헬스체크 실패
    - Connection timeout > 30초

  automatic_actions:
    - RDS Multi-AZ 자동 Failover 트리거 (30초~2분)
    - DNS 엔드포인트 자동 전환
    - 연결 풀 재설정

  manual_actions:
    - 트랜잭션 무결성 검증
    - 누락 트랜잭션 재처리 (WAL 기반)
    - 어플리케이션 재시작 (필요시)

  rollback:
    - 스탠바이 인스턴스 승격 취소
    - 마스터 인스턴스 복구 후 재동기화

# 시나리오 2: VAN 전체 장애
van_total_failure:
  detection:
    - 모든 VAN Circuit Breaker OPEN
    - 결제 실패율 > 50%

  automatic_actions:
    - 오프라인 모드 자동 전환
    - 키오스크 로컬 저장 활성화
    - 관리자 알림 (Level 2)

  manual_actions:
    - VAN사 연락 및 상황 확인
    - 현금 결제 우선 안내
    - 고객 대기열 관리

  recovery:
    - VAN 복구 확인 (Health Check PASS)
    - Circuit Breaker HALF_OPEN 전환
    - 대기 트랜잭션 순차 처리
    - 정상 모드 복귀

# 시나리오 3: 전체 시스템 장애 (리전 레벨)
region_failure:
  detection:
    - AWS 서비스 상태 페이지 확인
    - Route 53 헬스체크 연속 실패

  automatic_actions:
    - Route 53 DNS Failover (5분 TTL)
    - Cross-Region RDS 읽기 복제본 승격
    - S3 Cross-Region Replication 확인

  manual_actions:
    - DR 리전 인프라 완전 가동 확인
    - 데이터 정합성 검증
    - 고객 공지 발송
    - 키오스크 백엔드 URL 업데이트 (필요시)

  estimated_rto: 2시간
```
